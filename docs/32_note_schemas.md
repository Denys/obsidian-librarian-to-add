# 32 — Note Schemas

This document defines the first staged Obsidian note types generated by the librarian.

## Schema principles

- Every generated note is staged for human review.
- Every generated note keeps a source reference.
- Extracted action items stay separate from factual claims.
- Uncertainty is recorded instead of being hidden.
- Frontmatter should remain simple and machine-checkable.
- PDF-derived notes must preserve source hash, page count, extraction method, and page-level provenance before they can be treated as valid staged source notes.

## Source note

Represents one ingested Markdown or TXT file.

Required metadata:

- `type: source`
- `source_kind`
- `source_path`
- `project`
- `status: staged`
- `confidence`

Required sections:

- Summary
- Key claims
- Action items
- Open questions
- Links

## Planned PDF source note

Represents one ingested PDF source after the PDF compatibility phase is implemented.

Status: planned. See `docs/11_pdf_compatibility_plan.md`.

Required metadata:

- `type: source`
- `source_kind: pdf`
- `source_path`
- `source_hash`
- `project`
- `status: staged`
- `confidence`
- `page_count`
- `pdf_classification`
- `extraction_method`
- `ocr_enabled`

Required sections:

- Summary
- Key claims
- Action items
- Open questions
- Page evidence
- Extraction warnings
- Tables and figures
- Links

PDF-specific rules:

- `source_hash` is required so the staged note can be tied to an immutable PDF input.
- `page_count` is required so page-level provenance can be validated.
- `extraction_method` must identify whether content came from classifier-only metadata, Docling conversion, or OCR.
- `ocr_enabled` must be explicit and false unless OCR was deliberately requested.
- `Page evidence` must map extracted content back to page ranges or page anchors.
- `Extraction warnings` must include low-text, malformed, encrypted, OCR-needed, table-loss, figure-loss, or conversion-failure warnings when applicable.
- Table and figure outputs should be represented as linked sidecars/assets, not blindly flattened into prose.

## Planned PDF manifest sidecar

Represents deterministic metadata about a source PDF and its generated artifacts.

Status: planned. See `docs/11_pdf_compatibility_plan.md`.

Required fields:

- `source_path`
- `source_hash`
- `source_kind: pdf`
- `status`
- `page_count`
- `classification`
- `text_density`
- `extraction`
- `outputs`
- `provenance`

The manifest is not an Obsidian note by itself. It is a machine-checkable sidecar for reproducibility, validation, and future reprocessing.

## Atomic note

Represents one extracted concept.

Required metadata:

- `type: atomic`
- `source_path`
- `project`
- `status: staged`
- `confidence`

Required sections:

- Definition
- Why it matters
- Evidence
- Related

## Action extract

Captures follow-up work separately from factual notes.

Required metadata:

- `type: action_extract`
- `source_path`
- `status: staged`

## Uncertainty entry

Captures contradictory, incomplete, or low-confidence material for review.

Required metadata:

- `type: uncertainty`
- `source_path`
- `status: staged`

## Review report

Every ingest run should produce a review report with:

- processed files;
- skipped files;
- generated notes;
- warnings;
- validation results;
- write-safety events.

PDF-compatible review reports should additionally include:

- classified PDF candidates;
- skipped encrypted/malformed/scanned PDFs;
- extraction method and dependency status;
- generated PDF manifests and sidecars;
- page-level provenance validation results;
- OCR-needed/deferred warnings.
